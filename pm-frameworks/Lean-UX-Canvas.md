# Lean UX Canvas

**Category:** Validation (验证测试)

## Description

Integrates Lean Startup thinking into UX design—single canvas aligns problems, assumptions, MVP, and success metrics for rapid validation.

## The Canvas Sections

1. **Problem** - User pain points
2. **Hypotheses** - What we believe
3. **Assumptions** - Key beliefs to test
4. **MoSCoW** - Must/Should/Could/Won't have
5. **MVP** - Minimum viable product
6. **Test** - How we validate
7. **Metrics** - What we measure
8. **Insights** - What we learn

## When to Use

- UX design process
- Lean UX workshops
- Product design
- Team alignment

```markdown
# Lean UX 画布

## 概述

Lean UX 画布是 Jeff Gothelf 和 Josh Seiden 在《Lean UX》一书中提出的工具，旨在让整个团队在动手设计之前对齐问题框架。它将精益创业的「构建—测量—学习」循环与 UX 设计的「用户中心」原则结合，适合产品团队在 Sprint 启动时使用。

## 核心结构（8个模块）

| 编号 | 模块 | 关键问题 |
|------|------|----------|
| 01 | 业务问题 | 我们试图解决什么业务问题？现状是什么？目标是什么？ |
| 02 | 目标用户 | 我们在为谁设计？他们的核心需求、痛点和行为模式是什么？ |
| 03 | 用户结果与收益 | 用户希望用这个产品实现什么目标？成功对他们意味着什么？ |
| 04 | 解决方案 | 我们认为什么样的产品特性或体验能帮助用户达到上述结果？ |
| 05 | 核心假设 | 我们相信什么？哪些假设如果错了，整个方案就会失败？ |
| 06 | 假设排序 | 哪个假设风险最高？如果它是错的，影响最大？先验证哪个？ |
| 07 | MVP 实验方案 | 最小可行的实验是什么？用多少成本可以验证最高风险假设？ |
| 08 | 成功指标 | 什么数据能告诉我们假设被验证了？阈值是多少？ |

## 适用场景

- **Sprint/迭代启动会**：对齐团队对这个迭代要解决谁的什么问题
- **新功能立项前**：确认业务问题和核心假设，填不完整说明还没想清楚
- **跨职能团队对齐**：60分钟结构化对话，比会议纪要更有效
- **A/B测试设计前**：明确要验证的假设和成功条件
- **现有功能重新设计**：重新审视原有假设是否还成立
- **OKR拆解到具体实验**：从战略目标到验证实验的联接

## 关键原则

1. **全员共同填写**：价值在于填写过程中的对话，而非结果
2. **限时90分钟**：不追求完美，追求共识
3. **假设排序最重要**：「这个Sprint要验证的最高风险假设是什么？」这一句话比整张画布都重要
4. **每个Sprint结束后更新**：画布是活文档，而非一次性工具
5. **用便利贴填写**：物理或数字（Miro、FigJam）更适合修改和移动
6. **业务问题填不出就停**：如果无法用一两句话描述业务问题，说明功能还不成熟

## 推荐使用顺序

先用 Lean UX 画布（60分钟）对齐问题和假设，如果假设够明确、风险够高，再进入 Design Sprint（5天）做解决方案探索和原型验证。两者串联使用，能将 Design Sprint 效果提升 30–50%。

## 来源

- **提出者**：Jeff Gothelf & Josh Seiden
- **首次提出**：2013年《Lean UX》
- **灵感来源**：商业模式画布 + 精益创业方法论
- **应用公司**：Spotify、IBM、ThoughtWorks
```
