# POV Statement (POV 陈述)

**Category:** Definition (问题定义)

**Framework ID:** #14

**Source:** https://pmframe.works/framework-pov-statement

---

## Overview

Three-part format—User + Need + Insight—to precisely define core design problems, ensuring team alignment on "what we're solving."

---

## Core Formula

**[User]** needs to **[Need]** because **[Insight]**

---

## Three Components

| Component | Description |
|-----------|-------------|
| **User Persona** | Specific individual with real behavioral characteristics, not demographics like "25-35 year old women" |
| **Core Need** | Human-level verb describing emotional experience need, not functional features |
| **Unexpected Insight** | Surprising "why" discovered in research, not common sense |

---

## When to Use

- After user interviews for insight synthesis
- Before brainstorming (ensures alignment on the problem)
- When team has core problem disagreements
- Project kickoff phase
- Product direction pivot decisions
- Design Sprint initiation (Google Sprint Day 1)

---

## Case Study: IDEO x Shimano Coasting (2006)

- **Context:** 160M owned bikes in US, 90% of adults didn't ride
- **Key Insight:** Non-cyclists had dusty bikes in garages - cycling associated with childhood freedom, not adult gear complexity
- **POV:** "Adults interested in fitness and outdoor activities but who don't cycle need to experience cycling as effortless—because they associate cycling with childhood freedom, while adult cycling culture's equipment worship and professionalism intimidates them."
- **Three Design Principles:** No gear shifting, No special clothing required, Comfortable upright seating
- **Outcome:** Coasting concept bike with automatic shifting and coaster brake. Trek, Raleigh, Giant all launched products using this design.

---

## Five Common Mistakes

1. Embedding solutions in POV ("Users need an app that auto-syncs" is a feature, not POV)
2. User as demographic label instead of behavioral description
3. POV too broad ("Users need better experience" answers nothing)
4. Insight is common sense (real insight reveals psychological barrier root cause)
5. Only one POV (most research data supports 3-5 different POVs)

## Skill Download

**File:** `skills/pov-statement.md`

**Skill Content:**

```markdown
# POV Statement

Define a clear point-of-view statement using the format "[User] needs [need] because [insight]" to create an actionable problem frame that guides ideation.

## Steps

1. **Identify the user** — Specify a concrete user type based on research, not assumptions.
2. **State the need** — Express what the user needs as a verb phrase (not a solution). Focus on the goal, not a feature.
3. **Articulate the insight** — Add the "because" clause that reveals a surprising or non-obvious reason behind the need.
4. **Validate the statement** — Check that the POV is narrow enough to be actionable, broad enough to allow multiple solutions, and grounded in real evidence.
5. **Iterate** — Write 2-3 variations and select the one that best sparks "How Might We" questions.

## Output Format

A single POV statement in the format: "[User] needs [need] because [Insight]." Followed by 2-3 alternative phrasings and the rationale for selecting the final version.
```
