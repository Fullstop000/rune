# Empathy-Driven Roadmap

**Category:** Discovery (用户洞察)

## Description

Uses user pain severity as the core driver for roadmap prioritization—tackling the most severe user pains first.

## Core Principle

Prioritize product improvements based on the severity of user pain points, ensuring the team addresses the most significant user frustrations first.

## Process

1. **Identify pain points** - Gather user feedback and data
2. **Assess severity** - Evaluate impact on user experience
3. **Prioritize** - Rank by pain severity
4. **Plan** - Include in roadmap with clear outcomes

## When to Use

- Roadmap planning and prioritization
- User-centric product development
- Resource allocation decisions
- Communicating priorities to stakeholders

```markdown
# 同理心驱动路线图

**Empathy-Driven Roadmap**

让路线图每一个条目都承载用户洞察、机会假设和验证计划——从「功能清单」升级为可以说服工程师、销售和 CEO 的战略承诺文件。

## 核心结构

每个路线图条目包含三层：

**L1 · 用户洞察层**
谁在痛？为什么痛？有什么证据？
用户画像 + 痛苦场景 + 根本原因 + 证据强度

**L2 · 机会定义层**
如果解决了，影响有多大？假设可信度多高？
可量化目标 + 假设验证状态 + 影响力估算

**L3 · 解决方案层**
做什么？怎么验证成功？验证计划是什么？
产品方案 + 成功指标 + MVP 验证计划

## 使用步骤

1. **建立用户洞察库** — 定期访谈、产品反馈收集、支持工单分析
2. **为每个条目填写洞察层** — 回答：这个用户是谁？痛苦有多深？证据是什么？
3. **定义机会和成功指标** — 影响力 × 可信度 ÷ 开发成本 = 优先级分
4. **方案设计与验证计划** — 原型测试、灰度发布、Wizard of Oz 测试
5. **定期路线图 Review** — 更新假设验证状态，整合新洞察，重排优先级

## 路线图条目模板

**Q2 · 功能名称：智能分配建议**

**L1 · 用户洞察**
中型团队的项目经理（5–20人团队），在资源分配时花费大量时间手动平衡，因为没有可见的团队负载视图，常常导致部分成员过载。证据：访谈 12 位用户，9 位提及此痛点。

**L2 · 机会定义**
影响 38% 的付费用户。假设：如果提供智能分配建议，项目经理每周可节省 2–3 小时，留存率预计提升 8%。假设状态：概念测试已验证，未做功能验证。

**L3 · 解决方案**
在任务分配界面展示团队实时负载指标，并基于历史数据提供分配建议。MVP：先做只读的负载可视化，验证用户参与度后再加建议功能。

## 成功指标设计原则

- 每个条目应该有且只有一个「北极星指标」
- 避免把「功能使用率」当成成功指标
- 用户用了但痛苦没减少，不算成功
```
