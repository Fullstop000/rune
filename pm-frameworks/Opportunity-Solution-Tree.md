# Opportunity Solution Tree (机会解决方案树)

**Category:** Ideation (创意生成)

**Framework ID:** #29

**Source:** https://pmframe.works/framework-opportunity-solution-tree

---

## Overview

A four-layer tree structure (Outcome → Opportunity → Solution → Experiment) that forces product teams to explore user opportunities before jumping to solutions. OST breaks the "solution bias" cycle by requiring teams to identify opportunities from user research before generating any solutions.

---

## The Four Layers

| Layer | Description |
|-------|-------------|
| **Outcome** | The measurable goal (root) |
| **Opportunity** | User needs/discoveries from research |
| **Solution** | Ideas to address opportunities |
| **Experiment** | Tests to validate solutions |

---

## Key Steps

1. **Anchor to Outcome (30 min)** — Write ONE measurable outcome at the root, within team control. Example: "Increase 7-day retention from 30% to 45%."
2. **Continuous Interviewing (Weekly)** — Interview 1–2 real users weekly, ask about their experiences. Record exact quotes as opportunity nodes.
3. **Build Opportunity Layer (30 min/week)** — Add discoveries to the tree. Use users' actual language, not your interpretation. Don't prune yet.
4. **Generate Solutions (60 min)** — Select a high-priority opportunity, generate 3–5 candidate solutions per opportunity. Ask "What's the fastest way to validate this?"
5. **Design & Track Experiments (1–2 weeks each)** — Run A/B tests, Fake Door, Wizard of Oz, or prototype tests. Mark results on the tree.

---

## Case Study: Snyk — Developer security company (2020–2021)

- **Goal:** Increase developer adoption
- **Method:** Weekly interviews with 20 developers, building OST from discoveries
- **Key opportunities found:** (1) "I don't know my code has vulnerabilities until deployment"; (2) "Too many scan results, don't know what to fix first"; (3) "My team doesn't prioritize security"
- **Result:** IDE real-time scanning became Snyk's core differentiator
- **Valuation:** Grew to $8.5B

```markdown
# Opportunity Solution Tree

Structure product discovery by connecting a desired outcome to customer opportunities, potential solutions, and validation experiments. Use when you need to explore the solution space systematically rather than jumping to features.

## Steps

1. Define the desired **outcome** (business or product metric to move).
2. Identify **opportunities** — unmet customer needs, pain points, or desires discovered through research.
3. For each opportunity, brainstorm multiple **solutions** (at least 3 per opportunity).
4. For each solution, design a small **experiment** to test the riskiest assumption.
5. Run experiments, collect data, and prune the tree based on results.

## Output Format

A tree diagram with one outcome at the root, branching into opportunities, then solutions, then experiments. Each node includes a short label and confidence level (high/medium/low).
```
